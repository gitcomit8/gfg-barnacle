üîç The Bug

This module demonstrates a critical memory leak when passing complex state from Rust to JavaScript. The Rust code uses Box::into_raw() to hand ownership to JS, but the provided JS glue code never calls back into a dealloc function.

Result: Every user interaction creates ~100KB of leaked memory. The browser tab's memory usage will climb linearly until the WASM heap is exhausted, causing a RuntimeError: memory access out of bounds.

üîß The Technical Details

Rust Side (lib.rs):

#[wasm_bindgen] pub fn create_app_state(user_id: u32, username: String) -> usize { let state = AppState::new(user_id, username); let boxed_state = Box::new(state); // ‚ö†Ô∏è BUG: Returns raw pointer, ownership transferred to JS let raw_ptr = Box::into_raw(boxed_state); raw_ptr as usize } // ‚ùå MISSING FUNCTION - This should exist but doesn't! // pub fn free_app_state(state_ptr: usize) { ... }

JavaScript Side (index.js):

function createNewUser() { const statePtr = wasm.create_app_state(userId++, `user_${userId}`); currentStatePtr = statePtr; // ‚ö†Ô∏è BUG: We store the pointer but never free it! // The old pointer is lost forever, memory leaked! }

Why This Is Hard to Debug:

  • The interface looks like standard JS object integration
  • No immediate errors or warnings
  • Memory usage climbs gradually
  • Eventually crashes with "memory access out of bounds"
  • The missing dealloc function is easy to overlook

‚úÖ The Fix

To fix this bug, you need to:

  1. Uncomment the free_app_state function in src/lib.rs
  2. Call it in JavaScript before reassigning currentStatePtr
  3. Rebuild the WASM module
// Fixed JavaScript code: function createNewUser() { // ‚úÖ Free the old state before creating new one if (currentStatePtr !== null) { wasm.free_app_state(currentStatePtr); } const statePtr = wasm.create_app_state(userId++, `user_${userId}`); currentStatePtr = statePtr; }
